# Realtime DatabaseとFirestoreの違いについて

## Realtime Database
- データがひとつの大きなファイルにまとめて入っていて、誰かがデータを変更すると、見ている全員の画面に即座に反映される。とにかく「速く届ける」ことに特化しています。
- 「30代の人だけ表示」「福岡在住で絞り込む」といった条件での検索や並び替えが苦手。データ量が増えてくると、管理が難しくなる。

## Firestore
- データを種類ごとにまとめて保存できる。「ユーザー情報」「イベント情報」「マッチング結果」といった分け方ができるので、アプリが大きくなっても整理しやすい。
- 条件を組み合わせた検索も得意で、データが増えても動作が遅くなりにくい設計になっている。画面への反映もリアルタイムで行われる。

## compat版（旧来の書き方）とモジュール版（新しい書き方）

#### モジュール版
- 使う機能だけを必要なタイミングで読み込む書き方。アプリとして公開する場合に向いていますが、書き方がやや複雑。

#### compat版
- 昔からある書き方で firebase.firestore() のようにシンプルに書けます。学習中や、手軽に動かしたい場合はわかりやすい。

## 写真を保存するためのstorage（写真の保管庫）
- アップグレードを勧められる（いやだ）
- firestoreに写真そのものを文字として保存する方法があるらしい
写真を小さく縮小して、その写真を文字の形に変換してfirestorageに保存
- 表示する時は、その文字をそのまま写真として表示する（）
- Firestore の1人分のデータには1MBまでという上限がある
- 今回、写真は320pxに縮小するのでだいたい数十KB（上限の数十分の一）に収まり、まず問題ない。

## FirebaseのAPIキーはアップしても大丈夫？？
- FirebaseのAPIキーは「どのFirebaseプロジェクトに接続するか」を示す識別番号に過ぎない。ブラウザで動くアプリに埋め込む前提で設計されているので、誰が見ても問題ない値。
